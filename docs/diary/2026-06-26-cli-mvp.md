**Interpretation:** Start implementing the Outcome Engineering tool as an installable Python CLI managed by uv, using Typer for the command interface.
**Inferred intent:** Materialize the repo-native product graph idea into tooling agents can use to validate and navigate product structure.

### What I did

I added `/pyproject.toml`, `/uv.lock`, and a `src/outcome_engineering` package with a console command named `oe`. The first CLI commands are `oe validate`, `oe tree`, and `oe create-example`.

I moved the example generator into `/src/outcome_engineering/example.py`, kept `/scripts/create_example_product_graph.py` as a compatibility wrapper, and regenerated `/examples/delegation-product-graph`.

I added `/tests/test_graph.py` to cover validation of the generated example and the rule that experiments can only live under assumptions.

### Why

The framework needs a deterministic tool surface before agents can reliably maintain the product graph. Validation and tree navigation are the smallest useful foundation.

### What worked

`uv sync`, `uv run pytest`, `uv run oe validate examples/delegation-product-graph`, and `uv run oe tree examples/delegation-product-graph` all worked. The generated example now places `fake-connector-prototype` under the `operation-discovery-reduces-tool-overload` assumption.

### What didn't work

The original example had one experiment directly under a solution, which violated the clarified rule. I moved that experiment under the relevant assumption and made validation reject solution-level `experiments/`.

### What I learned

The CLI should treat relationship directories as edge declarations and marker files as node declarations. This makes validation simple and keeps the filesystem readable.

### What was tricky

The graph root is different from ordinary node directories because it can contain both `VISION.md` and `STRATEGY.md`. Nested node directories still need exactly one marker file.

### What warrants review

Review `/src/outcome_engineering/graph.py` for whether the current parent-child rules match the intended product graph. The next likely pressure point is whether PRDs should live directly under solutions only, or later support stories and code trace links.

### Future work

Add deterministic `oe new` commands for creating outcomes, opportunities, solutions, assumptions, experiments, and PRDs from templates. Add richer `oe challenge` checks once the structure is stable.
