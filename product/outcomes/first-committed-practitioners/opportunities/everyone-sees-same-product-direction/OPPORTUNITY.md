# I need product context available outside local setup

Solo builders, founder pairs, and small teams need the current product graph to be available in the product conversations and agent workflows where decisions actually happen.

When the graph only lives behind local repo and terminal workflows, it is harder to use as repeated product context. A founder may understand the graph and keep it updated, but co-founders, collaborators, or external agents cannot casually inspect it and work from the same product direction without help.

This matters because the graph is supposed to keep vision, strategy, outcomes, and current work connected. If product context cannot travel outside the local setup, the graph risks becoming private product memory instead of shared product context.

## Evidence

- Hypothesis: "I want my co-founders to see the same product graph I am using, without needing my local setup."
- Hypothesis: "I want to quickly revisit our product direction in a browser when we discuss what matters next."
- Hypothesis: "I want an external agent to understand the current product graph without needing a repo checkout."
- Hypothesis: Read-only access outside local setup may be enough before shared editing is necessary.

## Known / Unknown

- Known: The current product strategy says the graph should be visually usable and help users stay aligned with the vision.
- Known: The current product strategy says the graph should remain agent-legible.
- Known: The existing graph UI is local and requires running `oe serve`.
- Known: This need applies to the expanded ICP: solo builders, founder pairs, and small teams trying to keep product direction shared.
- Unknown: Which external product-context surface creates the most repeated value first: hosted human browsing, hosted agent context, or both together.
- Unknown: Whether read-only visibility creates enough alignment value before collaboration and editing features are added.
