# Agent Entry Points

Outcome Engineering should be useful before a team has a complete product graph.

The strategy is to let users start from messy product material, have an agent structure and challenge it, then grow a coherent graph around the parts worth keeping.

Agents should propose graph changes. Humans remain responsible for judgment and approval.

## Challenge Assumptions

Use this when the user has messy product thinking and wants to avoid building too long without validation.

Good inputs:

- A vague product idea.
- Meeting notes.
- A founder brain dump.
- A feature idea.
- A customer problem statement.
- A proposed solution.

The agent should help identify hidden assumptions, especially assumptions about value, usability, feasibility, and viability. It should distinguish what is grounded in evidence from what merely sounds plausible.

The useful output is not a perfect graph. The useful output is a clearer view of what must be true, what is uncertain, and what the team should learn next.

## Craft Vision And Strategy

Use this when the user needs the top of the graph to become clearer.

Good inputs:

- Founder or team context.
- Product notes.
- Existing positioning.
- A rough product direction.
- Unstructured conversation or meeting transcript.

The agent should help clarify the larger vision, the strategic focus, the outcomes that matter, and the tradeoffs implied by the strategy.

The useful output is a stronger product direction that can guide future graph nodes.

## Challenge Product Intent

Use this when the user already has something they may build or ship.

Good inputs:

- A PRD.
- A feature spec.
- An issue.
- An implementation plan.
- A code change.
- Acceptance tests.

The agent should challenge whether the work traces upward to an opportunity, outcome, strategy, and vision. It should ask whether the work is the highest-leverage thing to do next and whether the riskiest assumptions have been exposed.

The useful output is an intent check: what this work is supposed to change, why it matters, what evidence supports it, and where the graph has gaps.

## Grow The Graph Around The Work

The graph is durable product memory, not a form to fill out before work can begin.

Start with the user's actual material. Create or update only the graph nodes that make the work clearer, more traceable, or more actionable.

Prefer useful partial graphs over exhaustive structure. A small graph that helps a team make a better decision is better than a complete graph nobody trusts.
