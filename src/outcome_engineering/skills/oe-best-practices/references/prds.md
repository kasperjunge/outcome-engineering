# PRDs

A PRD connects the why to the what. It should not prescribe the how.

Solutions describe the chosen intervention. PRDs define the product requirements needed to deliver that intervention clearly.

Use these five rules when writing or reviewing PRDs:

1. Trace back to product intent.
   Connect the PRD to the solution, opportunity, outcome, strategy, and vision.

2. Define user-facing behavior.
   Say what must be true from the user or product perspective.

3. Decouple the why and what from the how.
   Explain why the work matters and what must be true. Do not include implementation details, architecture, library choices, database schemas, or code structure unless a technical constraint is product-critical.

4. Include acceptance criteria.
   Make success verifiable.

5. Preserve uncertainty.
   Mark unknowns as unknowns instead of pretending the PRD has resolved them.
